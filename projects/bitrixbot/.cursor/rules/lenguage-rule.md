# Cursor Rules — базовые правила для русскоязычного продукта

## Главный принцип
По умолчанию весь пользовательский интерфейс, тексты, подсказки, ошибки, empty states, onboarding, системные уведомления, письма и служебные сообщения должны быть написаны **на русском языке**.

## Исключения
Допускается оставлять **без перевода**:
- общепринятые термины: API, UI, UX, KPI, CRM, OKR, SQL, JSON, CSV, Excel, PDF, Dashboard, Widget, Login;
- названия брендов, сервисов, библиотек, фреймворков, языков программирования и продуктов;
- технические сущности, которые в отрасли обычно пишутся на английском, если перевод звучит неестественно или сбивает смысл.

## Правила локализации
- Пиши естественным, современным, понятным русским языком.
- Не используй буквальный машинный перевод, если по-русски есть более живой и короткий вариант.
- Интерфейс должен звучать так, как будто его изначально проектировали для русскоязычного пользователя.
- Предпочитай короткие и ясные формулировки.
- В кнопках и коротких действиях используй глаголы: «Сохранить», «Создать», «Удалить», «Открыть».
- Сообщения об ошибках должны объяснять проблему простыми словами и по возможности подсказывать, что делать дальше.
- Если есть выбор между англицизмом и понятным русским словом, выбирай русское слово.
- Не смешивай русский и английский в одной фразе без необходимости.
- Даты, числа, подписи, форматы и единицы измерения показывай в привычном для русскоязычного пользователя виде.
- Если создаёшь новый экран, компонент или фичу, сразу продумывай русский текст для всех состояний: default, loading, empty, error, success, confirm.

## UX-правила по умолчанию
- Все тексты в интерфейсе должны быть консистентны по терминам.
- Один и тот же объект везде должен называться одинаково.
- Текст должен быть ориентирован на пользователя, а не на внутреннюю техническую реализацию.
- Если термин новый или сложный, рядом должен быть понятный helper text.
- Любой интерфейс сначала делай понятным, потом красивым.

## Что проверять перед завершением задачи
- Нет ли лишнего английского в UI.
- Понятны ли кнопки, статусы и ошибки без объяснений разработчика.
- Одинаково ли названы сущности во всех местах.
- Не звучит ли текст сухо, роботизированно или слишком технически.
